# Научно-Исследовательская Работа

**Тема:** Исследование методов и технологий построения распределенной системы инвентарного учета с использованием блокчейн-ориентированного аудита

---

## Введение

### Актуальность темы

В современных условиях цифровизации бизнеса задачи инвентарного учёта приобретают критическую важность для организаций любого масштаба. Традиционные системы учёта сталкиваются с рядом фундаментальных проблем: риск человеческой ошибки при вводе данных, возможность незаметной фальсификации записей в централизованных базах данных, сложность отслеживания полной истории перемещений материальных ценностей. Особую остроту приобретает проблема доверия к данным аудита — даже при наличии журналов изменений администратор базы данных может внести правки задним числом, оставшись незамеченным.

Развитие технологии распределённых реестров (блокчейн) предлагает новый подход к обеспечению целостности данных. Применение криптографических хеш-цепочек для журналирования операций позволяет создать систему, в которой факт любой модификации исторических записей может быть детектирован с математической гарантией.

### Цель работы

Провести исследование существующих технологических стеков и архитектурных паттернов для создания системы инвентарного учёта, а также обосновать выбор гибридной архитектуры (реляционная СУБД + блокчейн-подобная структура журналирования) для обеспечения неизменяемости логов аудита.

### Задачи исследования

1. Выполнить анализ предметной области инвентаризации и выявить ключевые требования к системе.
2. Провести сравнительный анализ современных СУБД и бэкенд-фреймворков.
3. Исследовать теорию блокчейна и хеш-цепочек применительно к задачам аудита.
4. Выбрать средства фронтенд-разработки для обеспечения реактивности интерфейса.
5. Сформировать итоговый технологический стек для реализации проекта.

## Глава 1. Анализ предметной области и постановка задачи

### Характеристика объекта автоматизации

Инвентарный учёт представляет собой непрерывный процесс контроля наличия и перемещения материальных ценностей организации. Жизненный цикл предмета включает следующие этапы:

- **Поступление** — первичная регистрация предмета в системе с присвоением уникального идентификатора, указанием категории, стоимости, даты приобретения.
- **Размещение** — привязка предмета к конкретному зданию и помещению.
- **Перемещение** — фиксация изменения местоположения предмета между помещениями или зданиями.
- **Техническое обслуживание** — регистрация ремонтов, модернизаций, замен комплектующих.
- **Списание** — вывод предмета из эксплуатации с указанием причины (износ, утрата, моральное устаревание).

Ролевая модель пользователей системы включает:

- **Администратор** — полный доступ ко всем функциям системы, управление пользователями и настройками.
- **Редактор (Кладовщик)** — создание и редактирование записей о предметах, проведение операций перемещения и списания.
- **Аудитор** — доступ только на чтение, возможность просмотра журналов аудита и формирования отчётов.

### Функциональные требования

Система должна обеспечивать следующий базовый функционал:

1. **Ведение полного лога изменений** — каждая операция над предметом должна фиксироваться с указанием исполнителя, временной метки, типа операции, старых и новых значений полей.
2. **Поддержка загрузки медиа-контента** — возможность прикрепления фотографий предметов и сканов сопроводительных документов.
3. **Импорт и экспорт данных** — поддержка формата CSV для массовой загрузки номенклатуры и интеграции с устаревшими системами учёта.
4. **Поиск и фильтрация** — быстрый поиск предметов по различным атрибутам (инвентарный номер, категория, местоположение, статус).
5. **Генерация отчётов** — формирование сводных отчётов по состоянию инвентаря на заданную дату.

### Нефункциональные требования

Критически важные требования к качеству системы:

- **Целостность данных** — гарантия невозможности изменения прошлых записей аудита задним числом без обнаружения факта модификации. Это достигается за счёт применения хеш-цепочек.
- **Производительность** — время отклика системы не должно превышать 200 мс при работе с базой до 1 миллиона предметов.
- **Масштабируемость** — архитектура должна допускать горизонтальное масштабирование сервиса путём добавления новых экземпляров приложения.
- **Надёжность** — обеспечение доступности системы на уровне 99,9% в рабочее время организации.
- **Безопасность** — разграничение доступа на основе ролей, защита от распространённых веб-уязвимостей (OWASP Top 10).

## Глава 2. Исследование технологий серверной разработки и хранения данных

### Выбор языка программирования и среды исполнения

Современная разработка серверных приложений предлагает широкий выбор языков программирования. Основные альтернативы можно разделить на две группы:

**Интерпретируемые языки** (Python, JavaScript/Node.js, Ruby):
- Преимущества: высокая скорость разработки, богатые экосистемы библиотек, динамическая типизация.
- Недостатки: более низкая производительность выполнения кода, риски ошибок времени исполнения, сложности с поддержкой крупных проектов.

**Компилируемые языки** (Go, Rust, Java):
- Преимущества: высокая производительность, статическая типизация (отлов ошибок на этапе компиляции), эффективное использование ресурсов.
- Недостатки: более длительный цикл разработки, строгость синтаксиса.

Для проекта системы инвентарного учёта выбран язык **Go (Golang)**. Обоснование выбора:

1. **Производительность** — компилируемый код обеспечивает высокую скорость обработки запросов, что критично при работе с большими объёмами данных.
2. **Конкурентность** — встроенная поддержка горутин (goroutines) и каналов (channels) упрощает разработку многопоточных приложений.
3. **Статическая типизация** — снижает количество ошибок времени исполнения, улучшает поддерживаемость кода.
4. **Минимализм** — простой синтаксис, отсутствие излишних абстракций, быстрая компиляция.
5. **Экосистема** — богатый набор стандартных библиотек, активное сообщество, корпоративная поддержка от Google.

### Архитектурные паттерны бэкенда

При выборе фреймворка для Go рассматривались следующие варианты:

- **Gin** — популярный фреймворк с высокой производительностью, встроенной валидацией, поддержкой промежуточных обработчиков.
- **Echo** — минималистичный фреймворк с акцентом на расширяемость и простоту.
- **Chi** — лёгкий роутер, соответствующий стандарту `net/http`, без дополнительных зависимостей.

Выбор пал на **Chi/v5** по следующим причинам:

1. **Минимализм** — Chi предоставляет только необходимые функции маршрутизации, не навязывая дополнительную логику.
2. **Совместимость** — полная совместимость со стандартным пакетом `net/http`, что упрощает интеграцию со сторонними библиотеками.
3. **Отсутствие магии** — явная конфигурация маршрутов и промежуточных обработчиков облегчает понимание кода новыми разработчиками.
4. **Производительность** — эффективность маршрутизации сопоставима с более тяжёлыми фреймворками.
5. **Принцип KISS** (Keep It Simple, Stupid) — архитектура остаётся прозрачной и предсказуемой.

### Проблема хранения структурированных данных

Выбор системы управления базами данных является критическим решением. Рассматривались два основных подхода:

**Реляционные СУБД** (PostgreSQL, MySQL):
- Преимущества: строгая схема данных, транзакционность (ACID), мощные средства индексации, зрелость технологий.
- Недостатки: менее гибкая схема, сложности с горизонтальным масштабированием.

**Документоориентированные СУБД** (MongoDB, CouchDB):
- Преимущества: гибкая схема, удобное хранение иерархических данных, горизонтальная масштабируемость.
- Недостатки: отсутствие гарантий транзакционности (в некоторых реализациях), сложности со сложными запросами.

Для системы инвентарного учёта выбрана **PostgreSQL 16**. Обоснование:

1. **Строгая схема данных** — инвентарный учёт требует чёткой структуры (здания → комнаты → предметы), что естественно реализуется в реляционной модели.
2. **Транзакционность** — гарантии ACID критичны для финансовых операций и учёта материальных ценностей.
3. **Сложные связи** — поддержка внешних ключей, представлений, триггеров упрощает реализацию бизнес-логики.
4. **Расширяемость** — возможность добавления собственных типов данных, функций, операторов.
5. **Сообщество** — активная разработка, регулярные обновления, обширная документация.

### Подход к генерации кода для работы с БД

Традиционно разработчики используют ORM (Object-Relational Mapping) библиотеки для абстракции работы с базой данных. Однако ORM имеют ряд недостатков:

- Скрытые запросы (проблема N+1).
- Отсутствие гарантий типобезопасности запросов.
- Накладные расходы на маппинг объектов.

В проекте применён подход **Code-First с генерацией через sqlc**:

1. Разработчик пишет чистые SQL-запросы в отдельных файлах.
2. Инструмент `sqlc` анализирует запросы и генерирует типобезопасный Go-код.
3. Все запросы проверяются на этапе компиляции против схемы базы данных.

Преимущества подхода:

- **Типобезопасность** — несоответствие типов данных обнаруживается при компиляции.
- **Прозрачность** — разработчик видит точный SQL-код, выполняемый в базе.
- **Производительность** — отсутствие накладных расходов ORM.
- **Контроль** — полный контроль над каждым запросом к базе данных.

## Глава 3. Теоретические основы и применение блокчейн-технологий для аудита

### Основы технологии распределённых реестров

Блокчейн (цепочка блоков) — это структура данных, представляющая собой упорядоченную последовательность блоков, каждый из которых содержит ссылку на предыдущий блок. Ключевые компоненты блокчейна:

**Блок** — структурная единица, содержащая:
- Заголовок блока (метаданные).
- Список транзакций (данные).
- Хеш предыдущего блока.
- Собственный хеш.

**Хеш-функция** — криптографический алгоритм, преобразующий произвольные данные в строку фиксированной длины. Основные свойства:
- **Детерминированность** — одинаковые входные данные всегда дают одинаковый хеш.
- **Лавинный эффект** — минимальное изменение входных данных радикально меняет хеш.
- **Необратимость** — по хешу невозможно восстановить исходные данные.
- **Уникальность** — вероятность коллизии (совпадения хешей разных данных) крайне мала.

В контексте данной работы используется хеш-функция **SHA-256**, применяемая в стандартах безопасности и криптовалюте Bitcoin.

### Концепция «Аудит-чейна» в централизованной системе

Традиционный подход к ведению журналов аудита предполагает запись всех изменений в отдельную таблицу базы данных. Однако такой подход уязвим: администратор базы данных с правами суперпользователя может выполнить команды `UPDATE` или `DELETE` для модификации или удаления записей аудита, оставшись незамеченным.

**Решение на основе хеш-цепочки:**

Каждая запись аудита формируется следующим образом:

```
AuditRecord = {
    ID: уникальный идентификатор,
    Timestamp: временная метка,
    UserID: идентификатор пользователя,
    ActionType: тип операции,
    EntityType: тип сущности,
    EntityID: идентификатор сущности,
    OldValue: предыдущее значение (JSON),
    NewValue: новое значение (JSON),
    PrevHash: хеш предыдущей записи,
    CurrentHash: хеш текущей записи
}
```

**Алгоритм формирования цепочки:**

1. Для первой записи поле `PrevHash` устанавливается в нулевое значение.
2. Вычисляется хеш текущей записи без поля `CurrentHash`:
   ```
   HashInput = ID + Timestamp + UserID + ActionType + EntityType + EntityID + OldValue + NewValue + PrevHash
   CurrentHash = SHA-256(HashInput)
   ```
3. Запись сохраняется в базе данных.
4. Для следующей записи поле `PrevHash` устанавливается равным `CurrentHash` предыдущей записи.

**Математическое обоснование целостности:**

Если злоумышленник попытается изменить любую запись в прошлом:
1. Изменится содержимое этой записи.
2. Хеш изменённой записи (`CurrentHash`) станет другим.
3. Следующая запись содержит старый `PrevHash`, который больше не соответствует новому `CurrentHash` изменённой записи.
4. Нарушение цепочки обнаруживается при верификации.

Для успешной фальсификации необходимо пересчитать хеши всех последующих записей, что вычислительно сложно и требует значительного времени. При наличии большого объёма записей такая операция становится практически невыполнимой.

### Симуляция якорения (Anchoring)

Полноценное развёртывание распределённой блокчейн-сети (например, Hyperledger Fabric или Ethereum Private) требует значительных ресурсов и инфраструктуры. Для проекта предложена гибридная архитектура:

**Архитектурное решение:**

1. **Основное хранение** — все данные (предметы, пользователи, операции) хранятся в PostgreSQL для обеспечения высокой производительности.
2. **Хеш-цепочка аудита** — журнал аудита формируется с использованием хеш-цепочки внутри той же базы данных.
3. **Периодическое якорение** — корневой хеш цепочки (или хеш последней записи) периодически фиксируется во внешнем неизменяемом реестре.

**Варианты внешнего якорения:**

- **Публичные блокчейны** (Bitcoin, Ethereum) — транзакция с хешем записывается в блокчейн, обеспечивая абсолютную неизменность. Требует затрат на комиссии сети.
- **Приватные блокчейны** (Hyperledger Fabric, Corda) — развёртывание внутренней сети организации. Требует инфраструктуры, но даёт полный контроль.
- **Сервисы временного штампования** (RFC 3161) — использование доверенных служб временных меток.
- **Хранение в нескольких независимых системах** — дублирование хешей в разных базах данных или файловых хранилищах.

**Преимущества гибридного подхода:**

1. **Производительность** — основная нагрузка обрабатывается быстрой реляционной СУБД.
2. **Криптографическая гарантия** — хеш-цепочка обеспечивает детектирование модификаций.
3. **Гибкость** — возможность выбора стратегии якорения в зависимости от требований бюджета и безопасности.
4. **Масштабируемость** — архитектура допускает рост нагрузки без деградации производительности.

### Процедура верификации целостности

Для проверки целостности журнала аудита выполняется следующая процедура:

1. Извлекается первая запись аудита.
2. Вычисляется хеш записи по тем же правилам, что и при создании.
3. Сравнивается вычисленный хеш с сохранённым `CurrentHash`.
4. Извлекается следующая запись.
5. Проверяется, что `PrevHash` следующей записи равен `CurrentHash` текущей.
6. Повторяется для всех записей цепочки.

Любое несоответствие указывает на факт модификации данных. Результат верификации может быть оформлен в виде отчёта для аудитора.

## Глава 4. Исследование клиентских технологий и пользовательского опыта

### Парадигмы построения пользовательских интерфейсов

Современная веб-разработка предлагает несколько подходов к рендерингу пользовательских интерфейсов:

**SSR (Server Side Rendering):**
- HTML генерируется на сервере и отправляется клиенту.
- Преимущества: быстрая первоначальная загрузка, хорошая индексация поисковиками.
- Недостатки: меньшая интерактивность, нагрузка на сервер при каждом действии пользователя.

**SPA (Single Page Application):**
- Приложение загружается один раз, дальнейшие взаимодействия происходят через API.
- Преимущества: высокая интерактивность, ощущение нативного приложения, разделение ответственности между фронтендом и бэкендом.
- Недостатки: медленная первоначальная загрузка, сложности с SEO.

Для системы инвентарного учёта выбран подход **SPA на базе React 18**. Обоснование:

1. **Интерактивность** — пользователи работают с большими таблицами данных, фильтрами, формами ввода. SPA обеспечивает мгновенный отклик интерфейса.
2. **Компонентный подход** — переиспользование компонентов ускоряет разработку и поддерживает единообразие интерфейса.
3. **Экосистема** — огромное сообщество, множество готовых библиотек, активная поддержка от Meta.
4. **React 18** — новые возможности (concurrent rendering, automatic batching) улучшают производительность сложных интерфейсов.

### Управление состоянием и асинхронностью

Одна из ключевых проблем SPA — эффективное управление серверным состоянием. Традиционные подходы (Redux, MobX) требуют значительного объёма шаблонного кода для обработки асинхронных запросов.

**React Query (TanStack Query)** решает эту проблему:

1. **Кэширование** — автоматическое кэширование результатов запросов с настраиваемым временем жизни.
2. **Фоновое обновление** — данные обновляются в фоне при фокусировке окна или возврате на вкладку.
3. **Оптимистичные обновления** — интерфейс реагирует на действия пользователя мгновенно, до подтверждения сервером.
4. **Обработка ошибок** — встроенные механизмы повторных попыток и обработки ошибок.
5. **DevTools** — инструменты отладки состояния запросов.

Для инвентарной системы это означает:
- Мгновенный отклик при перемещении предметов (оптимистичное обновление).
- Автоматическое обновление списков при возврате пользователя на страницу.
- Минимизация сетевых запросов за счёт умного кэширования.

### Валидация данных и типизация

**TypeScript** обеспечивает сквозную типизацию всего приложения:

1. Типы данных определяются один раз и используются на всём стеке.
2. Ошибки несоответствия типов обнаруживаются на этапе компиляции.
3. Улучшенная автодокументированность кода и подсказки в IDE.

**Zod** — библиотека для валидации схем данных:

1. Определение схем валидации в декларативном стиле.
2. Автоматический вывод типов TypeScript из схем Zod.
3. Валидация данных, приходящих с сервера, перед использованием в приложении.
4. Гарантии соответствия данных между бэкендом и фронтендом.

Пример схемы предмета:
```typescript
const ItemSchema = z.object({
    id: z.string().uuid(),
    inventoryNumber: z.string().min(1),
    name: z.string().min(1).max(255),
    categoryId: z.string().uuid(),
    roomId: z.string().uuid().nullable(),
    status: z.enum(['active', 'maintenance', 'written_off']),
    createdAt: z.string().datetime(),
    updatedAt: z.string().datetime()
});
type Item = z.infer<typeof ItemSchema>;
```

### UI/UX решения

**Tailwind CSS** — утилитарный CSS-фреймворк:

1. Быстрая разработка интерфейсов через готовые классы.
2. Отсутствие необходимости придумывать имена классов.
3. Легко поддерживаемые стили за счёт ограничений утилитарного подхода.
4. Оптимизация размера конечного CSS-файла.

**Radix UI** — коллекция доступных примитивов компонентов:

1. Готовые реализации модальных окон, выпадающих списков, табов и других компонентов.
2. Полная доступность (ARIA-атрибуты, управление с клавиатуры).
3. Нестилевые компоненты — разработчик контролирует внешний вид.
4. Соответствие современным стандартам веб-доступности (WCAG).

Комбинация Tailwind CSS и Radix UI позволяет быстро создавать профессиональные интерфейсы с гарантированной доступностью для пользователей с ограниченными возможностями.

## Заключение

В ходе данного исследования был проведён комплексный анализ технологий и методов разработки системы инвентарного учёта. Основные результаты работы:

1. **Сформирован оптимальный технологический стек:**
   - Бэкенд: Go + Chi + PostgreSQL + sqlc
   - Фронтенд: React 18 + TypeScript + React Query + Zod + Tailwind CSS + Radix UI
   - Инфраструктура: Docker Compose для оркестрации сервисов

2. **Разработана теоретическая модель применения хеш-цепочек для аудита:**
   - Предложен механизм криптографического связывания записей журнала аудита.
   - Доказана возможность детектирования фальсификации данных даже при наличии прав суперпользователя базы данных.
   - Предложена гибридная архитектура с периодическим якорением хешей во внешних реестрах.

3. **Обоснованы архитектурные решения:**
   - Выбор компилируемого языка Go обеспечивает производительность и надёжность.
   - Минималистичный роутер Chi упрощает поддержку кода.
   - Генерация типобезопасного кода через sqlc устраняет класс ошибок времени исполнения.
   - Реактивный фронтенд на React обеспечивает высокий уровень пользовательского опыта.

4. **Доказано соответствие требованиям:**
   - Производительность: архитектура допускает обработку тысяч операций в секунду.
   - Масштабируемость: возможна горизонтальная масштабируемость бэкенд-сервисов.
   - Безопасность: ролевая модель, валидация данных, криптографическая защита аудита.
   - Целостность: хеш-цепочки гарантируют обнаружение любых модификаций истории.

Проведённое исследование создаёт прочный фундамент для перехода к этапу непосредственной реализации проекта. Выбранные технологии и архитектурные паттерны проверены временем, имеют активные сообщества и документацию, что снижает риски разработки и обеспечивает долгосрочную поддерживаемость системы.
