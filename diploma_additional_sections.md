3.4.5 Компоненты управления сущностями
Модуль manage реализует интерфейс для управления справочными данными системы: категориями инвентаря, зданиями и комнатами. Страница ManagePage организована с использованием вкладок (tabs), каждая из которых соответствует одному типу справочника. Переключение между вкладками осуществляется без перезагрузки страницы, состояние активной вкладки хранится в локальном состоянии компонента.

Вкладка категорий (CategoriesTab) отображает список категорий в виде сетки карточек. Каждая карточка содержит наименование категории и кнопки действий: редактирование и удаление. Добавление новой категории производится через встроенную форму (InlineForm), которая появляется в нижней части страницы при нажатии кнопки «Добавить категорию». Форма валидирует обязательное заполнение поля наименования. Мутации создания, обновления и удаления категорий выполняются через типизированные функции categoriesApi.create, categoriesApi.update, categoriesApi.delete. После успешного выполнения мутации происходит инвалидация кэша запроса ['categories'], что обеспечивает автоматическое обновление списка.

Вкладка зданий (BuildingsTab) реализует аналогичный паттерн для управления зданиями. Отличие заключается в структуре данных: каждое здание имеет два поля — наименование и адрес. Форма добавления/редактирования здания включает оба поля с соответствующей валидацией. Данные запрашиваются через buildingsApi.list с ключом ['buildings'].

Вкладка комнат (RoomsTab) имеет дополнительную зависимость от выбранного здания. Перед отображением списка комнат пользователь должен выбрать здание из выпадающего списка. Запрос комнат выполняется только после выбора здания (опция enabled в useQuery) с ключом ['rooms', buildingId]. Это предотвращает лишние запросы к API и обеспечивает контекстную фильтрацию данных. Форма добавления комнаты автоматически привязывает новую комнату к выбранному зданию при создании или обновлении.

Все три вкладки используют общий компонент InlineForm для отображения форм редактирования. Компонент принимает конфигурацию полей, значения, обработчики сохранения и отмены, а также статус ожидания мутации для блокировки кнопок во время выполнения запроса. Уведомления об успехе или ошибке операций отображаются через toast-контекст.

3.4.6 Компоненты управления пользователями
Модуль users предоставляет интерфейс администрирования учётных записей системы. Страница UsersPage отображает таблицу пользователей с возможностью создания, редактирования и удаления записей. Данные загружаются через хук useQuery с ключом ['users'] и функцией usersApi.list.

Каждая строка таблицы представляет карточку пользователя, содержащую полное имя, адрес электронной почты и-badge роли (администратор или редактор). Кнопки действий включают редактирование (доступно для всех пользователей, включая текущего) и удаление (скрыто для текущей учётной записи для предотвращения случайного самоудаления; дополнительная защита реализована на уровне бэкенда).

Диалог создания/редактирования пользователя (UserFormDialog) реализован с использованием библиотеки react-hook-form и валидации через zod. Схема валидации различается для режимов создания и редактирования: при создании требуются все поля (full_name, email, password, role), при редактировании — только full_name и role. Пароль не отображается и не может быть изменён через форму редактирования из соображений безопасности.

Форма отображается в модальном окне с затемнением фона. При отправке данных вызывается соответствующая мутация usersApi.create или usersApi.update. После успешного выполнения происходит инвалидация кэша ['users'] и закрытие диалога.

Подтверждение удаления реализовано через отдельное модальное окно с явным указанием имени удаляемого пользователя. Это предотвращает случайное удаление записей. Мутация удаления usersApi.delete выполняется только после подтверждения пользователем.

Отображение ролей локализовано: вместо технических значений 'admin' и 'editor' отображаются переведённые названия через функцию t('users.role_admin') и t('users.role_editor').

3.4.7 Компоненты настроек приложения
Модуль settings реализует страницу персональных настроек пользователя. Страница SettingsPage имеет минималистичный интерфейс, разделённый на три логические секции: профиль, язык интерфейса и выход из системы.

Секция профиля отображает информацию о текущем авторизованном пользователе, полученную из контекста аутентификации useAuth(). Отображаются полное имя, адрес электронной почты и роль пользователя. Данные берутся непосредственно из объекта user контекста, без дополнительных запросов к API.

Секция выбора языка позволяет переключать язык интерфейса между русским и английским. Переключение выполняется через метод i18n.changeLanguage() библиотеки i18next. Выбранный язык подсвечивается визуально (активная кнопка имеет стиль primary). Выбор языка сохраняется в localStorage через настроенный плагин i18next и восстанавливается при следующей загрузке приложения.

Кнопка выхода вызывает функцию logout() контекста аутентификации, которая очищает токен доступа из localStorage и перенаправляет пользователя на страницу входа через navigate('/login'). Стилизация кнопки выполнена в деструктивном цвете (border-destructive) для визуального выделения критического действия.

Страница настроек не выполняет запросов к API, все данные получаются из локального состояния приложения (контекст аутентификации, состояние i18n). Это обеспечивает мгновенную реакцию интерфейса на действия пользователя.
